Agentic Workflow

Spec-driven development
for coding agents.

Having an AI agent is one thing. Getting agents and people to collaborate is another.

A shared, living memory for product teams and their agents, so intent, decisions, and the current spec stay in sync as the work evolves.

01 / Problem
The real problem

Agents don't fix your collaboration
problems. They inherit them.

👥

Product, Design & Engineering already struggle to stay aligned

  • PRDs go stale as the build evolves
  • Requirements change with no record of what or why
  • Intent drifts from PRD → conversation → code
  • Decisions, and their rejected alternatives, get lost
+ agent
🤖

The agent joins a team that already loses the thread

So it loses it too: re-deriving intent, re-litigating settled decisions, drifting from the spec, exactly like a teammate dropped in with no context.

An agent doesn't remove these problems. It runs straight into them and adds new ones.

don't go away.

03 / Solution
The solution

One shared memory.
For agents and people.

Agentic Workflow keeps intent, decisions, and the current spec as living documents in the repo, the same source of truth your team and your agents both read before they act.

It fixes collaboration on two fronts at once: human ↔ agent, the agent works from current intent instead of a stale PRD, and human ↔ human, fewer re-alignment meetings and clarification cycles because the doc is always live.

Two steps. One config file. Works in any repo, with any team, with any agent runtime.

# 1. install the skills (skills.sh) npx skills add antonyjclements/agentic-workflow # 2. scaffold this repo, in your agent /aw-init # what aw-init writes write: AGENTS.md write: CLAUDE.md write: docs/workflow/config.yml write: docs/decisions/index.yml write: docs/features/index.yml write: docs/learnings/index.yml agentic workflow installed
04 / Solution
The entry point

AGENTS.md: the map every agent reads first.

Claude Code, Codeium, and Windsurf all read AGENTS.md at session start. Agentic Workflow installs a canonical AGENTS.md that tells agents:

  • Where living specs live and how to read them
  • How to route requests to the right skill
  • What size of change needs what level of ceremony
  • When to capture decisions, learnings, and solutions
  • How to use the config to find custom enterprise skills

It's versioned, upgradeable, and the same across every repo that uses this workflow.

What's inside AGENTS.md
  • 01Operating principles, read before changing, scope discipline, preserve user work
  • 02Task triage, match ceremony to change size
  • 03Spec-driven model, where each artifact lives and its lifecycle
  • 04Step routing, config-driven skill selection and handoff contract
  • 05Capture checkpoints, when to log decisions, learnings, solutions
  • 06Safety & expectations, git rules, testing, documentation, final summary

One canonical file, versioned and identical across every repo that adopts the workflow.

06 / Artifacts
Decisions in depth

Immutable records.
Derived navigation.

Decision files are never edited after creation. When a decision is superseded, a new file is created that references the old one. The chain is permanent and auditable.

The index and summaries are derived, regenerated by aw-refresh-decisions without touching the immutable records. You get current navigation without rewriting history.

Immutable records

use-agents-skills.md original decision
keep-repo-local.md supersedes the original

Derived navigation

index.yml current status and links
summaries/workflow.md regenerated by tag

Why immutability matters

  • Links never break, old docs still point to valid files
  • Full audit trail of how thinking evolved
  • Agents can see why a decision was superseded, not just that it was
  • No accidental rewriting of history under time pressure

07 / Workflow
The pipeline

Every step hands off
to the next one.

Each skill outputs the artifact path that becomes the next skill's input. The chain is traceable from PRD to shipped PR.

📥

Import / Author PRD aw-import-prd · aw-create-prd

Persist the source artifact first → docs/product/prds/*.md

💬

Brainstorm aw-brainstorm

Resolve ambiguity → outputs a living spec.

📋

Living Spec aw-create-spec

Durable intent at docs/features/<feature>/spec.md: when requirements are already clear.

🗺️

Plan aw-plan · aw-review-doc

Temporary map at docs/features/<feature>/plan.md, reviewed before work.

🎫

Tickets aw-create-tickets

Linear / Jira / custom. Each traces back to spec, plan, decisions.

⚙️

Work aw-work · aw-debug

From a ticket, spec, plan, or bare request. Ticket-first cold-start works.

🚢

Review, Commit, PR, CI aw-review-code · aw-commit-push-pr · aw-monitor-pipeline

PR body includes test policy, acceptance coverage, decisions, residual risks.

Quality backstops · wrap Work → PR, not steps in the chain
test_policy: acceptance criteria → tests/checks aw-review-spec: catches code-vs-spec drift aw-check-workflow-compliance: verifies the contract on the shipping path
08 / Workflow
Ceremony that scales

Not every change
needs the full path.

The built-in task triage matches workflow depth to change size. Agents use judgment, and the compliance check catches under-triaging after push.

Trivial

Edit directly. No spec, no plan, no review, no compliance.

Small fix

Debug if needed. Targeted tests. Update docs only if behavior changes.

Feature

Living spec → plan → work → acceptance tests → review gates → PR.

High-risk

Full path. Decision log mandatory. Missing spec or test evidence is a blocker.

09 / Skills
The skill system

31 skills. One global install.
Every agent runtime.

Skills are SKILL.md files that agents read as workflow instructions. They're installed globally into ~/.agents/skills/ and symlinked to Claude Code, Codeium, and Windsurf automatically.

Each skill is self-contained: trigger conditions, workflow steps, rules, and handoff contract. Agents invoke them by name. The runtime routes the call.

Any skill can be replaced per-repo via docs/workflow/config.yml without touching the global install.

10 / Config
Extensibility

Enterprise-ready.
Zero config to get started.

docs/workflow/config.yml lets teams replace any bundled skill with an enterprise-specific one, without touching AGENTS.md or the global install.

All values are optional. Blank means use the bundled default. Most repos never need to change anything.

  • Custom ticket creation (Linear, Jira, internal)
  • Enterprise Slack research skill
  • Custom commit message format and templates
  • PR title/body templates from markdown files or URLs
  • Human reviewer lists for specs and plans
  • Post-PR CI provider (CircleCI, GitHub Actions, manual)
  • Implementation test policy per repo

When config shape changes, aw-upgrade migrates it, dry-run first, timestamped backup before apply, conflict detection.

Override one route:
workflow: steps: commit_push_pr: skill: enterprise-pr
blank = default

Every key is optional. Set one only to override the bundled skill for that route. The full schema lives in the README.

11 / Knowledge
The feedback loop

Context accumulates.
It doesn't evaporate.

Capture checkpoints are built into AGENTS.md, so decisions, corrections, and solutions get written down as they happen, for the agent and for the next person who opens the repo.

📌

Decisions

When ambiguity is resolved or a trade-off is made. Immutable. Referenced by future agents automatically.

Trigger: resolve ambiguity, choose between options
🎓

Retrospectives

When the user corrects, redirects, or says "this should have worked differently." Prevents the same mistake next session.

Trigger: correction, interruption, post-completion redirect
💡

Solutions

When a non-trivial bug, integration issue, or repeatable problem is solved. Becomes reusable team knowledge.

Trigger: non-trivial problem solved, solution is repeatable
How it compounds, for agents and people
Decision or correction → Captured in the repo → Agent reads it next session → Teammate reads it next sprint

Observed building SkillStack: live documentation cut confusion among people, not just agents, fewer re-alignment meetings and fewer "wait, what are we building again?" cycles.

12 / Maintenance
Upgradeable by design

The workflow stays
current across repos.

🔢

Version tracking

Every installed repo gets .agentic-workflow-version. The installer reads aw-version.txt: one file to bump for a release.

☁️

Remote install

The --remote flag fetches the latest installer and all skills directly from GitHub, or an internal mirror via --source-url. No local clone needed to update an existing install.

🔄

Config migration

aw-upgrade migrates old config.yml shapes safely: dry-run preview, timestamped backup, conflict detection, no silent data loss.

13 / Verdict
Assessment

Who it's for.
What it delivers.

✓ Where it excels

  • Cross-functional teams who lose intent between PRD, chat, and code
  • Multi-session projects where context compounds over weeks
  • Teams that write specs and log decisions, workflow makes them shared and live
  • Enterprise environments needing custom skill routing

⚠ Honest caveats

  • Discipline-amplifier, not discipline-creator, won't force teams to keep specs alive
  • Triage boundaries rely on agent judgment, not hard rules
  • Higher ceremony than raw prompting for one-off scripts

"A lone agent delivers a fraction of the value, because the collaboration problems are still there."

The leverage isn't a smarter agent. It's agents plus shared memory, live documentation, and decision tracking, all working the same source of truth your team does.

The outcome isn't better agents. It's smoother collaboration, agent to human, and human to human.
