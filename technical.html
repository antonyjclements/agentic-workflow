01 / Premise
Start here

An agent is only as good as
the context you give it.

Yes. It's just markdown an agent reads. No runtime, no library, no language binding. That part is obvious.

The interesting part is the systems design layered on top of plain documents, to make an agent's context durable, current, and compounding instead of re-derived from scratch every session.

And that context isn't only for the agent. It's the same memory your team reads, so the real problem is collaboration: human ↔ agent, and human ↔ human. From here, this deck is the engineering that makes that shared memory work.

The shift

Most agent tooling optimizes the prompt. This optimizes the repo: the persistent context your team and your agents both read before they act, and write after they learn.

Get that right and raw model capability stops being the bottleneck. Continuity does.

The bet: durable, well-structured context beats a cleverer prompt, every session, and more so over time.
02 / Roadmap
What we'll explore

Battle-tested patterns,
pointed at an agent's context.

No new magic. We took four patterns every engineer already trusts and aimed them at how an agent remembers and works. Here's the tour.

🔁

Event sourcing for decisions

An immutable decision log with derived, regeneratable projections: current state without rewriting history.

🔌

Dependency injection for skills

Routing as data: the call site names a role, config binds the implementation. Swap any step without forking.

🔣

State machines for artifacts

Every artifact type has a lifecycle. The load-bearing invariant: mutable intent, immutable history.

⚖️

A determinism boundary

An explicit line between what's guaranteed in code and what's left to agent judgment, and why it's there.

The payoff: memory that compounds across sessions. Strap in.

03 / Substrate
What actually executes

The agent is the interpreter.

A classic framework has a deterministic runtime as its execution substrate. This operating model's substrate is an LLM agent. The "instructions" are natural language; the "interpreter" reasons over them probabilistically, so the context it reasons over is the whole game.

Agent (interpreter) Reads instructions, reasons, decides which skill to invoke, writes artifacts. Probabilistic.
▲ reads & follows ▲
Instruction layer AGENTS.md + SKILL.md files, routing rules, lifecycle rules, handoff contracts. Prose.
▲ reads & writes ▲
Artifact layer docs/: specs, decisions, learnings, standards, plans. The repo's persistent memory.
— — — no coupling — — —
Application code Your actual program. Untouched by the workflow's existence. Governed by it only via the agent.

The only deterministic code in the entire system is the installer (a shell script) and two small helper scripts. Everything else is interpreted by the agent.

05 / Artifact lifecycle
Each artifact is a state machine

Lifecycle rules, formalized.

Artifact types are more than folders. Each has a defined lifecycle the operating model enforces. Five durable knowledge types carry state; plans are temporary. The distinction between mutable intent and immutable record is the load-bearing idea.

Spec · living
draft → active ⟲ updated

Mutated in place whenever behavior changes. Always reflects current truth. One per feature.

Decision · immutable
active → superseded

Never edited after creation. Supersession creates a new file referencing the old. Append-only log.

PRD · historical
imported → promoted → archived

Preserved as input, never treated as living truth. archived ⇒ removable; git is the archive.

Learning · correction-driven
captured → indexed → applied next session

Written on correction. Read at session start. Closes the mistake-repetition loop.

Core invariant: intent is mutable, history is immutable. Specs are rewritten freely; decisions are never rewritten, only superseded. Everything else derives from these two rules.
06 / Composition
Skills compose by contract

Skills are pure functions
over artifacts.

Every skill obeys the same contract: take an artifact path (or identifier) as input, produce the next artifact path as output. The output of one step is the typed input of the next.

This is what makes the pipeline traceable end-to-end, and what makes any step replaceable. A custom enterprise skill is valid as long as it honours the same input/output contract.

Contract: accept the same handoff artifact, read relevant config, return the expected artifact path / ID, and report unsupported behavior explicitly.
the handoff chain, output ⊳ input
aw-import-prd⊳docs/product/prds/x.md
aw-create-spec⊳docs/features/x/spec.md
aw-plan⊳docs/features/x/plan.md
aw-create-tickets⊳TICKET-123
aw-work⊳branch + diff
aw-commit-push-pr⊳github.com/…/pull/7
aw-monitor-pipeline⊳green CI

A later agent can re-enter the chain at any node, e.g. cold-start from just TICKET-123: by loading the linked upstream artifacts.

Here aw-brainstorm handles the ambiguous intake; with clear requirements, aw-create-spec writes the spec directly.

What the repo remembers afterward

  • features/mfa/spec.md: living intent, kept current
  • decisions/…totp-over-sms.md: immutable, forever
  • learnings/…clock-skew.md: auto-applied next session

What expires

  • features/mfa/plan.md: deleted; the job is done
  • prds/mfa.md: archived; git keeps the history
Six months later, an agent picking up MFA-103 reads the spec, sees why TOTP was chosen, and won't re-litigate the clock-skew bug, none of which was in any chat window.
08 / Decision model
Immutability + projection

Event sourcing,
for decisions.

The decision registry borrows the event-sourcing pattern: the decision files are the immutable event log; the index and summaries are read projections rebuilt from that log.

  • Source of truth: individual decision .md files, append-only, never mutated
  • Projection: index.yml + summaries/<tag>.md: fully regeneratable
  • Rebuild: aw-refresh-decisions recomputes projections from the log

This resolves the tension every decision log hits: "records must never change" vs "we need to know which are current." Currency lives in the projection; truth lives in the log.

log → projection

Immutable log

use-feature-dirs.md record says: active
use-single-plan.md supersedes: use-feature-dirs
aw-refresh-decisions
rebuilds

Derived projection

use-feature-dirs current status: superseded
use-single-plan current status: active

The original file still says status: active in its own frontmatter, and that's correct. The projection holds current state; the record holds what was true when written.

09 / Routing
Indirection layer

config.yml is dependency injection
for skills.

Every configurable step resolves through one indirection: read workflow.steps.<step>.skill; if blank, use the bundled default. This is dependency injection, the call site names a role, the config binds the implementation.

  • 15 core steps, the sequential pipeline
  • 13 auxiliary helpers, invoked from within steps
  • Blank = default, zero config to start
  • Bound = override, swap in Linear, Jira, enterprise commit, internal Slack search…

Because routing is data, not code, an org customizes the entire workflow without forking a single skill, and aw-upgrade migrates the config shape forward with dry-run + backup + conflict detection.

role → implementation binding
create_tickets → aw-create-tickets blank = default
commit_push_pr → enterprise-pr override
research_slack → corp-slack-search auxiliary

Resolution rule

The agent calls the configured skill when one is bound; otherwise it uses the bundled default for that role.

Liskov for skills: any override must be substitutable for the default, same contract in, same contract out.
10 / Determinism
Where determinism lives

Draw the line between
deterministic and probabilistic.

A mature operating model is explicit about which parts are guaranteed and which are interpreted. Putting the boundary in the wrong place is how these systems fail.

Deterministic, code

  • Installer: which files are written, where
  • Version stamping & the upgrade migration
  • Symlink wiring across runtimes
  • Config schema validation & backup
  • Smoke test assertions

Shell + Ruby. Tested. Reproducible. No model involved.

Probabilistic, agent

  • Which triage tier a change belongs to
  • When to capture a decision or learning
  • How a spec is written and updated
  • How the configured test policy is applied
  • Whether two decisions actually conflict
  • Skill selection for ambiguous requests

Interpreted from prose. Judgment, not guarantee.

Design rule: anything that must be guaranteed (install integrity, no data loss on upgrade, immutability of records) is pushed into deterministic code. Everything requiring judgment is left to the agent, with review gates as the backstop.
Quality gates: test_policy guides implementation style (TDD, BDD, characterization-first, or explicit verification); aw-review-spec checks spec drift; aw-check-workflow-compliance checks process adherence.
11 / Guarantees
What it does & doesn't promise

Honest guarantees.

Guaranteed (by construction)

  • Zero runtime coupling, removable without breaking the app
  • Decision records are never silently mutated
  • Upgrades back up before they migrate; no silent data loss
  • Context that is captured is available to the next session
  • The same install works across stacks and runtimes

Not guaranteed (requires discipline)

  • That specs stay current, the model amplifies discipline, it can't create it
  • That every decision gets logged, capture is prompted, not forced
  • That TDD/BDD is perfectly followed, policy guides the agent and review exposes misses
  • That triage is always right. It's agent judgment
  • Deterministic output for the same input, the substrate is probabilistic

An operating model can't compile your discipline into existence. It can make discipline cheap to follow and expensive to skip, and then make the gap visible when you skip it anyway.

12 / Principles
Design principles, distilled

The five ideas underneath.

1 · No runtimeThe agent is the interpreter; the repo supplies only intent and memory.
2 · Memory over orchestrationPersist context in the repo; let the agent orchestrate against it.
3 · Mutable intent, immutable historyRewrite specs; never rewrite decisions, only supersede and project.
4 · Routing is dataBind roles to implementations in config; customize without forking.
5 · Determinism where it must be guaranteedCode for integrity; agent judgment for everything else.

Not a framework like BMAD or SpecKit. Those drive a prescribed process. An operating model defines how work is done and lets the agent run it.
